## 2.1 Streaming 流式响应

**问题**：普通对话等 AI 全部生成完才一次性渲染，用户等待 5-30 秒无反馈。

**做法**：
- 在 `AiGateway` 接口新增 `streamChat()` 方法，返回 `AsyncIterable<StreamChatChunk>`
- 在 `defaultAiGateway` 实现三厂商流式：
  - **OpenAI 兼容**（OpenAI / DeepSeek / Qwen / GLM / Kimi / Ollama / Custom）：`stream: true` + `stream_options.include_usage`，按 SSE `data:` 行解析 delta
  - **Anthropic**：监听 `content_block_delta` / `message_delta` 事件
  - **Gemini**：`streamGenerateContent?alt=sse`，最后一帧含 `usageMetadata`
- `ChatPanel` 改造发送流程：先调用 `parseUserInput` 解析意图（非流式），若是普通对话则切到 `streamChat` 路径，逐 chunk 追加到消息内容
- 通用 SSE 行解析器 `readSseDataLines`：按 `\n` 切分，遇 `data:` 前缀收集 payload，空行作帧边界
- 注意：`parseUserInput` / `runAgentAnalysis` 必须等完整 JSON（结构化解析无法逐 token 用），仍走 `generateJson`

**关键文件**：`src/services/aiGateway.ts`、`src/components/app-shell/ChatPanel.tsx`

## 2.2 Markdown 渲染增强

**问题**：Agent 输出的 rawMarkdown 以纯文本渲染，缺少语法高亮和表格美化。

**做法**：

**关键文件**：`src/components/common/Markdown.tsx`（新建）、`src/components/app-shell/ChatPanel.tsx`

## 2.3 Token 用量统计

**问题**：`ChatCompletionResponse.usage` 字段从未展示，用户无法感知消耗。

**做法**：
- 在 `aiGateway.ts` 维护 `sessionUsage` 单例：`promptTokens` / `completionTokens` / `totalTokens` / `callCount` / `byModel`
- 提供 `getSessionUsage()` / `resetSessionUsage()` / `recordUsage()` 接口
- `ChatPanel` 底部显示"本次会话累计消耗"（token 数 + 调用次数）
- 按 `${providerLabel}/${modelName}` 维度汇总，便于多模型对比
- 重置时机：用户切换会话 / 主动点击"清空会话" / 应用重启

**关键文件**：`src/services/aiGateway.ts`、`src/components/app-shell/ChatPanel.tsx`

## 2.4 聊天历史自动压缩（滑动窗口策略）

**问题**：消息无限堆积在 prompt 中，可能超过模型 token 上限导致截断或报错。

**做法**：
- 新建 `src/services/tokenEstimator.ts`，提供：
  - `estimateTokens(text)`：粗略估算（字符数 / 2，中文为主场景的折中值）
  - `estimateMessageTokens(msg)`：单条消息 token 数（含 role 开销约 4 token）
  - `buildSlidingWindowMessages(messages, maxTokens)`：从最新消息向前累加，截取不超过阈值的最近 N 条；始终保留最后一条
- 在 `ChatPanel.runSendPipeline` 中，调用 `streamChat` 前对当前会话历史应用滑动窗口（阈值 6000 tokens，预留 2048 给新生成）
- **本地消息不变**：滑动窗口只影响发送给 AI 的 messages 数组，被截断的旧消息仍在本地完整保留可查看

**关键文件**：`src/services/tokenEstimator.ts`（新建）、`src/components/app-shell/ChatPanel.tsx`

## 2.5 错误重试与消息编辑

**问题**：消息发送失败仅显示提示，无法针对单条消息重试；无法编辑已发送消息后重新发送。

**做法**（在 `ChatPanel.tsx` 中实现，无需新建文件）：
- `retryMessage(msgId)`：重置该消息状态为 `sending`，清空其后所有消息，重新走 `runSendPipeline` 发送流程
- `startEdit(msgId)` / `cancelEdit()`：把最后一条 user 消息内容回填到输入框，用户编辑后重新发送（删除原消息及其后所有消息）
- `rerunAgentJob(runId)`：针对失败的 Agent 任务，重新触发 `runJobNow`
- `MessageItem` 组件根据消息角色与状态渲染对应操作按钮：
  - 用户消息：`showRetryOnUser` / `showEditOnUser`
  - 失败消息：`showRetryOnError`
  - Agent 执行消息：`showRerunAgent`

**关键文件**：`src/components/app-shell/ChatPanel.tsx`

## 2.6 多会话管理（轻量下拉方案）

**问题**：只有一个聊天流，用户对话 + Agent 巡检 + 系统通知全混在一起，无法按主题分隔。

**做法**：
- **复用** `chat.ts` 中已定义但未使用的 `Conversation` interface（`id` / `title` / `createdAt` / `updatedAt`）
- `ChatMessage` 新增可选字段 `conversationId?: string`（向后兼容）
- `appStore` 新增状态：`conversations: Conversation[]`、`activeConversationId: string | null`
- 新增 4 个会话操作方法：
  - `createConversation()`：生成「新会话 YYYY-MM-DD HH:mm」标题，加入列表并切换为活跃
  - `switchConversation(id)`：切换活跃会话
  - `renameConversation(id, title)`：重命名
  - `deleteConversation(id)`：删除会话及其所有消息；删活跃会话时自动切换到第一个剩余会话，无剩余则新建空白会话
- `initApp` 自动迁移：若 `conversations` 为空但 `messages` 非空，创建默认「主会话」并给现有消息补 `conversationId`
- `addMessage` 自动写入当前 `activeConversationId`，Agent 定时任务消息也路由到活跃会话
- 新建 `ConversationSwitcher` 组件：ChatPanel 头部右侧的会话下拉 + 「新对话」按钮 + 列表项悬停显示「重命名」「删除」操作 + 确认弹窗
- 消息列表按 `activeConversationId` 过滤显示
- Rust 端 ALLOWED_KEYS 同步新增 `"conversations"` 数据键

**关键文件**：`src/domain/chat.ts`、`src/services/localStore.ts`、`src-tauri/src/main.rs`、`src/store/appStore.ts`、`src/components/app-shell/ConversationSwitcher.tsx`（新建）、`src/components/app-shell/ChatPanel.tsx`

## 2.7 Tauri 代理 AI API 调用（非流式）

**问题**：前端 `fetch()` 直连 AI API，`resolveApiKey()` 把 API Key 拉回前端内存后才发送请求，运行时 Key 仍会短暂出现在 JS 堆中。

**做法**：
- **Rust 端**新增 `call_ai_api` async 命令：
  - 定义 `CallAiApiArgs` / `CallAiApiResponse` 结构体
  - 通过 `keyring::Entry::new(KEYRING_SERVICE, &api_key_ref)` 从系统凭据读 API Key
  - 按 provider 分流：
    - **OpenAI 兼容**（OpenAI / DeepSeek / Qwen / GLM / Kimi / Ollama / Custom）：`POST ${baseUrl}/chat/completions`，`Authorization: Bearer ${key}`
    - **Anthropic**：`POST ${baseUrl}/messages`，`x-api-key` + `anthropic-version` header，拆分 system 与 user/assistant 消息
    - **Gemini**：`POST ${baseUrl}/models/${modelName}:generateContent?key=${apiKey}`，转换 messages 为 `contents` + `parts` 格式
  - 支持 `jsonMode`：OpenAI 兼容请求体附加 `response_format: { type: "json_object" }`
- **前端** `aiGateway.ts` 新增 `callAiApiViaRust` 函数封装 `invoke("call_ai_api")`
- `callOpenAICompatible` / `callAnthropic` / `callGemini` 三个非流式函数优先走 Rust 代理（`isTauri() && model.apiKeyRef` 条件下），失败回退到原 `fetch` 路径
- **回退策略**：
  - 业务错误（"未配置 API Key" / "API 错误"）直接抛出
  - 基础设施错误（invoke 失败、网络异常）`console.warn` 后回退到前端 fetch，保证浏览器开发环境仍能工作
- `generateText` / `generateJson` / `testConnection` 与 `runAgentAnalysis`（用 `generateJson`）均自动受益
- **流式 `streamChat` 暂保留前端 fetch + SSE 直连**（非本项关注点）

**关键文件**：`src-tauri/src/main.rs`、`src/services/aiGateway.ts`

## 技术选型小结

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 2.4 压缩策略 | 滑动窗口 | 零额外 AI 调用、无延迟、实现简单 |
| 2.6 会话 UI | 轻量下拉 | 改动小、不新增左侧页面、符合"侧栏已满"现状 |
| 2.7 代理范围 | 仅非流式 | 流式需 Tauri event 改造，投入产出比暂不划算 |
| 2.7 回退机制 | 自动回退 fetch | 保证浏览器开发环境与 Tauri 生产环境双兼容 |

## Phase 2：数据层建设（对标 a-stock-data）

> 日期：2026-07-08
> 来源：对照 [a-stock-data](https://github.com/simonlin1212/a-stock-data) 的 13 源 40 端点数据架构
> 目标：让 AI 从"只能看价格"升级为"全维度分析师"

### P0-1：K 线数据接入（腾讯财经）

**问题**：AI 分析持仓时只能看到当前价格，无法做技术面判断。

**方案**：
- Rust 端新增 `get_kline(symbol, period, count)` 命令
- 数据源：腾讯财经 HTTP API（`qt.gtimg.cn`），无封 IP 风险
- 返回最近 N 根 K 线（开/高/低/收/量），可带 MA5/MA10/MA20
- `agentRunner.runAgentJob` 在拼装上下文时注入 K 线摘要（如"近 20 日均线走势：..."）
- 注意：首次只做日 K，后续扩展到 60 分/30 分 K

**关键文件**：`src-tauri/src/main.rs`（新增 command）、`src/services/marketData.ts`（新增前端接口）、`src/services/agentRunner.ts`（注入上下文）

### P0-2：PE/PB/市值/换手率（腾讯财经）

**问题**：AI 无法判断股票"贵不贵"，没有估值锚点。

**方案**：
- Rust 端新增 `get_fundamentals(symbol)` 命令
- 数据源：腾讯财经基本面 API
- 返回：PE（静态/动态/TTM）、PB、总市值、流通市值、换手率、52 周最高/最低
- Agent 分析 prompt 增加"估值维度"段落，注入 PE 分位、PB 等

**关键文件**：同 P0-1

### P0-3：行情数据源 fallback 机制

**问题**：仅新浪单一数据源，挂了就全瘫。

**方案**：
- 在 `get_batch_quotes` 中增加腾讯财经为第二数据源
- 新浪失败时自动回退到腾讯
- `MarketDataSource` 类型扩展为支持优先级和多源列表
- 用户可在数据源设置页调整优先级

### P1-1：北向资金流向（东方财富）

**问题**：AI 看不到外资动向，缺少重要的资金面信号。

**方案**：
- Rust 端新增 `get_north_bound_flow(date)` 命令
- 数据源：东方财富 HTTP（有封 IP 风险，需实现 `em_get` 级限流：>=1s + 0.1~0.5s 随机抖动）
- 返回：当日北向净流入/流出、累计净买入、分市场明细（沪/深）
- Agent 分析 prompt 增加"资金面维度"段落
- Rust 端增加东财限流器（`tokio::time::sleep` + 随机延迟 + 连接复用）

**关键文件**：`src-tauri/src/main.rs`（限流器 + 新 command）、`src/services/agentRunner.ts`

### P1-2：股东户数变化（东方财富）

**问题**：无法判断筹码集中度。

**方案**：
- Rust 端新增 `get_shareholder_count(symbol)` 命令
- 返回：最新股东户数 + 较上期变化 + 历史趋势
- Agent 分析 prompt 中注入"筹码集中度"信号

### P1-3：一致预期 EPS（同花顺）

**问题**：AI 只能看历史 PE，无法做前瞻估值。

**方案**：
- Rust 端新增 `get_consensus_eps(symbol)` 命令
- 返回：当年/次年一致预期 EPS、机构覆盖数
- Agent 分析可计算前向 PE / PEG

## Phase 3：AI 分析能力升级

> 目标：单模型单维度 → 多模型多维度辩论

### 3-1：多维度分析报告

**问题**：当前 `AgentAnalysisOutput` 只有 `marketOverview` / `risks` / `opportunities`，维度单一。

**方案**：
- 扩展 `AgentAnalysisOutput` 结构，增加 5 维度评分：
  - `technicalScore`（技术面 1-10）：趋势/均线/量价
  - `fundamentalScore`（基本面 1-10）：估值/盈利/成长
  - `capitalFlowScore`（资金面 1-10）：主力/北向/筹码
  - `sentimentScore`（情绪面 1-10）：热度/题材/舆情
  - `riskScore`（风险面 1-10）：回撤/集中度/事件
- 每个维度附带 1-2 句判断依据
- 综合建议基于 5 维度加权得出（用户可配置权重）
- 聊天面板中渲染为维度和雷达图卡片

**关键文件**：`src/domain/agent.ts`、`src/services/aiGateway.ts`（prompt 重写）、`src/components/app-shell/ChatPanel.tsx`（渲染维度卡片）

### 3-2：多模型辩论机制

**问题**：单一模型可能存在偏见，用户无法感知观点可靠性。

**方案**：
- 用户可指定 2-3 个模型参与辩论（从已启用的模型中选择）
- 每个模型独立分析同一持仓，输出独立结论 + 置信度
- 汇总阶段：
  - 标注一致性（全部看多 / 多数看多 / 分歧 / 多数看空 / 全部看空）
  - 标注共识点和分歧点
  - 给出综合建议（附各模型意见分布）
- 默认使用 1 个模型（不增加费用），用户可在 Agent 设置中开启辩论模式
- 注意：调用次数翻倍 = 费用翻倍，UI 需明确提示

**关键文件**：`src/domain/agent.ts`（新增 `DebateResult` 类型）、`src/services/aiGateway.ts`（新增 `runAgentAnalysisDebate`）、`src/pages/AgentPage.tsx`（辩论模式开关）

### 3-3：调研流水线模板

**问题**：Agent 分析缺少标准化 SOP，每次分析范围不固定。

**方案**：
- 在 Agent 创建时新增"分析策略"选项：
  - **快速估值**（~30s）：价格 → 一致预期 EPS → 前向 PE/PEG
  - **标准巡检**（~60s，默认）：技术面 + 基本面 + 风险面
  - **深度调研**（~120s）：机构覆盖 → 估值 → 概念 → 资金 → 龙虎榜 → 解禁 → 两融
  - **同类对比**（~60s）：同行业多只股票横向排列
- 不同策略用不同的系统 prompt，调不同的数据端点
- 取决于数据层 P0/P1 的建设进度

**关键文件**：`src/domain/agent.ts`（`analysisStrategy` 字段）、`src/services/aiGateway.ts`（各策略 prompt 模板）、`src/pages/AgentPage.tsx`（策略选择 UI）

### 3-4：智能提醒引擎升级

**问题**：当前提醒只有价格条件，触发维度太少。

**方案**：
- 扩展 `AlertRule` 类型支持：
  - 技术面条件：MA 突破/跌破、MACD 信号、RSI 超买超卖
  - 资金面条件：北向连续 N 日流入/流出、股东户数骤变（变化率 > X%）
  - 事件条件：财报发布、解禁到期、大宗交易
  - 组合条件：上述条件的 AND/OR 组合
- `agentRunner.runAgentJob` 分析完成后检查提醒条件
- 触发时通过 `notification.ts` 推送系统通知

**关键文件**：`src/domain/agent.ts`、`src/services/agentRunner.ts`、`src/services/notification.ts`

## Phase 4：体验增强

### 4-1：K 线图嵌入聊天

**关键文件**：`src/components/common/MiniKlineChart.tsx`（新建）、`src/components/app-shell/ChatPanel.tsx`

### 4-2：市场情绪仪表盘

**问题**：用户打开应用后无法一眼感知市场整体状况。

**方案**：
- Dashboard 新增"市场温度"卡片区域：
  - 涨停/跌停/炸板 家数（需数据层 P2 涨停板数据）
  - 北向资金实时净流向（需数据层 P1-1）

  - 行业热力排行

**关键文件**：`src/components/dashboard/MarketTemperature.tsx`（新建）、`src/pages/DashboardPage.tsx`

### 4-3：自然语言选股

**问题**：用户只能跟踪已有持仓，无法发现新机会。

**方案**：

- 调用数据层多因子筛选接口
- 返回候选列表（表格形式），附带简要分析
- 用户可一键将候选股加入观察列表

**前提**：数据层 P0 + P1 全部完成

### 4-4：研报/公告 AI 摘要

**方案**：
- Agent 巡检时自动拉取持仓股的最新公告/研报标题
- 如有新内容，AI 做 3-5 条要点压缩
- 在聊天框推送："你持仓的 XX 股今天发布了 Q2 财报，AI 摘要：..."

## 技术选型补充

| 决策点 | 选择 | 理由 |
|--------|------|------|
| K 线数据源 | 腾讯财经 HTTP | 无封 IP 风险、纯 HTTP 协议、Rust reqwest 直接代理 |
| 基本面数据源 | 腾讯财经 HTTP | 同上，PE/PB/市值/换手率接口稳定 |
| 东财数据访问 | Rust reqwest + 限流器 | 统一技术栈，不引入 Python；限流器用 `tokio::time::sleep` + `rand` 随机抖动 |
| 是否引入 Python | 暂不引入 | P0/P1 数据都可用 HTTP 获取；P2 季报/F10 需要时再评估 |
| 辩论模式默认值 | 关闭（单模型） | 不增加用户默认费用；高阶用户手动开启 |
| K 线图库 | lightweight-charts | TradingView 开源版，轻量、专业、支持叠加技术指标 |
