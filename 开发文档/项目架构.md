# AI 炒股 Agent 系统 — 项目架构

> 版本：v0.1 | 更新：2026-07-07
> 形态：Windows 桌面 GUI 应用（Tauri + React + TypeScript）

---

## 1. 整体架构

```text
┌─────────────────────────────────────────────────────────────┐
│                    Tauri 桌面应用壳                          │
│  ┌──────────────────────────────────────────────────────┐   │
│  │            Rust 主进程（src-tauri/src/main.rs）       │   │
│  │  - 创建窗口 / 系统通知                                │   │
│  │  - 本地文件读写（AppData 目录，key 白名单）            │   │
│  │  - API Key 安全存储（Windows Credential Manager）     │   │
│  │  - 行情 HTTP 代理（避免浏览器 CORS）                  │   │
│  └────────────────────┬─────────────────────────────────┘   │
│                       │ invoke('command_name', args)         │
│  ┌────────────────────▼─────────────────────────────────┐   │
│  │              React 前端（src/）                       │   │
│  │  - UI 渲染（React + Tailwind + shadcn/ui）            │   │
│  │  - 业务逻辑（Zustand store + services）              │   │
│  │  - AI 调用（aiGateway，前端直连 LLM API）             │   │
│  │  - Agent 调度（前端 setInterval，v0.2 迁移到 Rust）   │   │
│  └──────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**关键设计**：
- Rust 端只做"系统级能力"（文件/凭据/通知/行情代理），不包含业务逻辑
- 业务逻辑全部在前端（TypeScript），便于迭代
- AI 调用由前端直连 LLM API（v0.2 计划迁移到 Rust 代理，见优化建议 2.7）

## 3. `src/` 前端源码详解

### 3.1 入口文件

| 文件 | 作用 |
|---|---|
| `main.tsx` | React 应用入口，挂载到 `#root`，导入全局样式 |
| `App.tsx` | 顶层路由组件：初始化应用 → 判断首次启动（Onboarding）→ 渲染 AppShell + 当前页面 |
| `vite-env.d.ts` | Vite 环境类型声明 |

### 3.2 `domain/` — 领域模型（纯类型定义，无逻辑）

| 文件 | 作用 |
|---|---|
| `account.ts` | 账户领域模型：`Account`（id/name/cumulativePrincipal/cashBalance/timestamps） |
| `position.ts` | 持仓领域模型：`Position`（symbol/name/quantity/avgCost/currentPrice/marketValue/unrealizedPnl/...） |
| `trade.ts` | 交易记录领域模型：`Trade`（type=BUY/SELL/symbol/quantity/price/amount/note/timestamp） |
| `agent.ts` | Agent 领域模型：`AgentJob`（triggerType=interval/fixed_time/condition，tradingHoursOnly，scope，nextRunAt）+ `AgentRun`（执行记录）+ `AlertRule`（风险提醒规则）+ 标签/颜色映射常量 |
| `chat.ts` | 聊天领域模型：`ChatMessage`（role/type/content/metadata）+ `ParsedDraft`（AI 解析用户输入后的结构化草稿，含 intent/queriedSymbols/tradingHoursOnly 等）+ `Conversation` |
| `ai.ts` | AI 领域模型：`AiModelConfig`（provider/modelName/baseUrl/apiKey/isDefault）+ `ChatCompletionRequest/Response` + 厂商列表常量 |
| `config.ts` | 应用配置领域模型：`AppConfig`（theme/appMode/tradingHoursOnlyByDefault）+ `MarketDataSource` + 默认值常量 + `TRADING_HOURS`（A 股交易时段 9:30-11:30/13:00-15:00） |
| `memory.ts` | 记忆领域模型：`Memory`（type=preference/rule/agent_note/note，title/content/timestamps） |

### 3.3 `services/` — 业务服务层

| 文件 | 作用 |
|---|---|
| `aiGateway.ts` | AI 网关：统一封装多厂商 LLM 调用（OpenAI 兼容/Anthropic/Gemini），提供 `generateText`/`generateJson`/`streamChat`/`testConnection`；包含 `PARSE_SYSTEM_PROMPT`（用户输入→ParsedDraft 的解析提示词）和 `runAgentAnalysis`（Agent 分析提示词）；`parseUserInput` 解析用户意图 + queriedSymbols |
| `marketData.ts` | 行情服务：`getQuote`/`getBatchQuotes`（Tauri 真实时走 Rust 代理，浏览器 dev 时走 fallback）；`matchSymbolByName`（名称→代码，15 只硬编码 + 代码 fallback）；`inferMarketByCode`（按代码前缀推断市场 SH/SZ/BJ/HK/US）；`searchSymbol`；15 秒内存缓存 |
| `agentRunner.ts` | Agent 执行引擎：`runAgentJob`（刷新行情→拼装上下文→调 AI→写聊天消息→更新执行记录）；`calculateNextRunAt`（计算下次执行时间，支持 tradingHoursOnly 对齐交易时段）；`isJobDue`（判断是否到期，tradingHoursOnly 时检查交易时段） |
| `scheduler.ts` | Agent 调度器：`AgentScheduler` 类，前端 `setInterval` 每 30 秒 tick 一次，遍历 jobs 调 `isJobDue`，到期触发 `onRun` 回调。v0.2 计划迁移到 Rust 主进程 |
| `tradingCalendar.ts` | 交易日历工具：`isTradingDay`（排除周末）、`isWithinTradingHours`（9:30-11:30/13:00-15:00）、`alignToNextTradingSession`（把任意时间对齐到下一个交易时段开始，处理周末跳转） |
| `localStore.ts` | 本地存储服务：通过 Tauri `invoke('read_app_json'/'write_app_json')` 读写 AppData 下的 JSON 文件，按 key 白名单（account/positions/trades/messages/...）分文件存储 |
| `notification.ts` | 通知服务：封装 Tauri `show_notification` 命令，弹出系统通知 |

### 3.4 `store/` — 状态管理

| 文件 | 作用 |
|---|---|
| `appStore.ts` | 全局状态（Zustand）：聚合所有领域数据（account/positions/trades/messages/agentJobs/agentRuns/alerts/memories/models/config/dataSources/accountSnapshots）+ 所有业务 action（deposit/withdraw/addPosition/sellPosition/addTrade/addAgentJob/runAgentNow/addAlert/addMemory/updateModel/setConfig/resetAssets/clearAllData/initApp/recordSnapshot）。是整个应用的"单一数据源" |

| `read_api_key(key_ref)` | 从系统凭据读取 API Key |
| `delete_api_key(key_ref)` | 删除系统凭据中的 API Key |

## 5. 数据流

### 5.1 用户发送消息流程

```text
用户输入 → ChatPanel.send()
  │
  ├─ 阶段1：parseUserInput(model, text, context)
  │    └─ AI 返回 ParsedDraft { intent, queriedSymbols, requiresConfirmation, summary, ... }
  │
  ├─ 如果 requiresConfirmation=true
  │    └─ 渲染确认卡片，等用户点"确认"
  │    └─ 用户确认 → applyDraft(draft) → 调用 store.addPosition/addAgentJob/...
  │
  └─ 如果 requiresConfirmation=false（query/chat 意图）
       ├─ 如果有 queriedSymbols → getBatchQuotes(symbols) → 拼入 systemPrompt
       └─ streamChat(model, messages) → 逐 token 追加到消息内容
```

### 5.2 Agent 定时任务流程

```text
AppShell 启动 → new AgentScheduler(onRun=runAgentJob) → setInterval(30s)
  │
  ├─ tick() → 遍历 jobs → isJobDue(job, now)?
  │    └─ 检查 enabled + nextRunAt <= now + tradingHoursOnly 时检查交易时段
  │
  └─ 到期 → runAgentJob(job, ctx)
       ├─ 1. getBatchQuotes(positions.symbols) 刷新行情
       ├─ 2. 拼装上下文（accountSummary/positions/marketData/memories）
       ├─ 3. runAgentAnalysis(model, context) 调 AI
       ├─ 4. 写入聊天消息（role=agent, type=agent_run）
       ├─ 5. 更新 AgentRun 状态（success/failed）
       └─ 6. calculateNextRunAt(job) 更新 nextRunAt
```

### 5.3 本地存储流程

```text
前端 store.setXXX() → localStore.saveXXX(data)
  └─ invoke('write_app_json', { key: 'positions', content: JSON.stringify(data) })
       └─ Rust validate_key('positions') → 白名单校验
       └─ 写入 AppData/positions.json（先 .tmp 再 rename 原子写）
```

## 6. 关键设计决策

### 6.1 为什么 Agent 调度在前端（v0.1）

- 开发快，无需 Rust 端复杂的状态同步
- 应用打开时即可工作，满足 MVP 需求
- **限制**：应用关闭后 Agent 停止；v0.2 计划迁移到 Rust 主进程 + Tauri 事件推送

### 6.2 为什么 AI 调用在前端直连

- 多厂商 API 差异大，前端 SDK 更丰富
- 流式响应（SSE）前端处理更直接
- **限制**：API Key 在前端内存中；v0.2 计划 Rust 代理（见优化建议 2.7）

### 6.3 为什么用 JSON 文件而非 SQLite

- 用户无需安装数据库
- 数据易备份/导出/人工检查
- MVP 阶段数据量小，JSON 足够
- 后续可平滑迁移到 SQLite

### 6.4 为什么行情走 Rust 代理

- 浏览器直接调新浪行情会被 CORS 拦截
- Tauri webview 通过 Rust 端 `reqwest` 调用，无 CORS 问题
- 浏览器 dev 模式下走 fallback（可能失败，符合预期）

### 6.5 为什么用轻量级行情注入而非 Function Calling

- 所有 OpenAI 兼容模型都能用（含 DeepSeek）
- 改动小（3-4 文件 vs 6-8 文件）
- AI 不需要支持 Function Calling
- 详见 [优化建议.md](./优化建议.md) 的 MCP 决策记录

## 8. 开发命令

```bash
# 开发模式（Tauri + Vite 热更新）
npm run tauri:dev

# 仅前端 dev（行情/AI 可能受 CORS 限制）
npm run dev

# 打包桌面应用（生成 exe）
npm run tauri:build

# TypeScript 类型检查 + 前端构建
npm run build
```

**Windows 注意**：PowerShell 执行策略可能阻止 npm 脚本，需先执行
`Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass`，或改用 cmd。
