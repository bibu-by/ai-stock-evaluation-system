# AI 炒股 Agent 系统 — 优化建议

> 日期：2026-07-04  
> 来源：对照热门开源项目 [NextChat](https://github.com/ChatGPTNextWeb/NextChat)（88k+ Stars）进行差距分析  
> 定位：在保持本项目「股票投研 + Agent 监控」核心差异化的基础上，补齐通用 AI 聊天体验

---

## 1. 与 NextChat 的定位差异（不可丢的核心优势）

NextChat 是一个**通用 AI 聊天客户端**，它的职责是"让你和任何模型聊天"。  
本项目的定位是**垂直领域的个人投资工作站**，差异在于：

| 能力 | NextChat | 本项目 |
|------|----------|--------|
| 多模型聊天 | ✅ | ✅ |
| 持仓/账户管理 | ❌ | ✅ |
| Agent 定时巡检 | ❌ | ✅ |
| 自然语言解析持仓 | ❌ | ✅ |
| 风控条件提醒 | ❌ | ✅ |
| 投资记忆库 | ❌ | ✅ |
| 行情数据接入 | ❌ | ✅ |
| API Key 系统凭据加密 | ❌ | ✅ |

**结论**：以上差异化能力是护城河，绝不能弱化。优化应在保持这些能力的基础上进行。

## 2. 优化建议分级

### 🔴 Tier 1 — 直接影响用户体验，尽快做（1-2 周）

- [x] #### 2.1 Streaming 流式响应

- **NextChat 做法**：AI 回复逐 token 出现，不等待完整响应
- **现状**：等 AI 全部生成完才一次性渲染，Agent 分析时用户可能等 10-30 秒无反馈
- **建议**：
  - 在 `aiGateway.ts` 中增加 `streamChat()` 方法，使用 fetch `ReadableStream` + SSE 解析
  - ChatPanel 中逐 token 追加到消息内容
  - 注意：`generateJson()` 类调用（解析持仓、Agent 分析）仍需等待完整 JSON，但可先显示"正在分析…"动画

- [x] #### 2.2 Markdown 渲染增强

- **NextChat 做法**：LaTex 公式、Mermaid 图表、代码高亮、表格
- **现状**：Agent 输出的 rawMarkdown 大概率以纯文本渲染，缺少语法高亮和表格美化
- **建议**：
  - 引入 `react-markdown` + `remark-gfm`（表格/任务列表）
  - 引入 `react-syntax-highlighter`（代码块着色）
  - Agent 输出中的数字表格（持仓对比、盈亏变化）用 Markdown 表格天然渲染

- [x] #### 2.3 Token 用量统计

- **NextChat 做法**：不直接显示，但这是 AI 工具的基本素养
- **现状**：`aiGateway.ts` 的 `ChatCompletionResponse` 已包含 `usage` 字段，但从未展示
- **建议**：
  - 在聊天面板底部显示"本次会话 Token 用量"
  - 在模型设置页增加"累计用量"（按模型汇总）
  - 可选：按厂商价格估算费用

- [x] #### 2.4 聊天历史自动压缩

- **NextChat 做法**：超出上下文窗口时自动裁剪/摘要旧消息
- **现状**：消息无限堆积在 prompt 中，可能超过模型 token 上限导致截断或报错
- **建议**：
  - 估算当前会话 token 数（粗略按 `字符数/2` 或引入 `tiktoken`）
  - 超过阈值时对早期消息做摘要压缩（调用一次 AI 做摘要）
  - 或使用滑动窗口保留最近 N 轮对话
- **实现**（2026-07-07）：采用滑动窗口策略。新增 `src/services/tokenEstimator.ts` 提供 `estimateTokens`（字符数/2）与 `buildSlidingWindowMessages`（从最新向前累加，阈值 6000 tokens）。`ChatPanel.runSendPipeline` 在调用 `streamChat` 前对当前会话历史应用滑动窗口，被截断的旧消息仍在本地完整保留可查看。

- [x] #### 2.5 错误重试与消息编辑

- **NextChat 做法**：消息发送失败可重试、可编辑已发送消息后重新发送
- **现状**：错误仅显示提示，无法针对单条消息重试
- **建议**：
  - 失败的消息旁边显示"重试"按钮
  - 用户可编辑自己最后一条消息并重新发送
  - Agent 执行失败后支持"重新运行"
- **实现**（2026-07-07）：已在 `ChatPanel.tsx` 中完整实现。包含 `retryMessage`（重置状态、清空后续消息、重新走发送流程）、`startEdit/cancelEdit`（编辑最后一条 user 消息后重新发送）、`rerunAgentJob`（重新运行失败的 Agent 任务）。`MessageItem` 根据 `showRetryOnUser` / `showEditOnUser` / `showRetryOnError` / `showRerunAgent` 渲染对应操作按钮。

### 🟡 Tier 2 — 显著提升产品力（2-4 周）

- [x] #### 2.6 多会话管理

- **NextChat 做法**：左侧会话列表，可新建/切换/删除多个对话
- **现状**：只有一个聊天流，所有消息（用户对话 + Agent 巡检 + 系统通知）混在一起
- **建议**：
  - 引入 `Conversation` 概念（替代当前的单一消息列表）
  - 支持新建会话、切换会话、重命名、删除
  - Agent 自动消息仍写入当前活跃会话，或独立写入"Agent 日志"会话
- **实现**（2026-07-07）：采用轻量下拉方案。复用 `chat.ts` 中已定义但未使用的 `Conversation` interface。`ChatMessage` 新增 `conversationId` 字段，`appStore` 新增 `conversations` / `activeConversationId` 状态与 `createConversation` / `switchConversation` / `renameConversation` / `deleteConversation` 四个方法。`initApp` 自动迁移：无会话时创建默认「主会话」并给现有消息补 `conversationId`。`addMessage` 自动写入活跃会话 id，Agent 定时任务消息也路由到当前活跃会话。ChatPanel 头部右侧新增 `ConversationSwitcher` 组件（会话下拉 + 「新对话」按钮 + 重命名/删除确认弹窗）。消息列表按 `activeConversationId` 过滤显示。Rust 端 ALLOWED_KEYS 同步新增 `"conversations"` 数据键。

- [x] #### 2.7 Tauri 代理 AI API 调用

- **NextChat v2.15.4**：通过 Tauri Rust 层 fetch LLM API，前端不直接接触 API Key
- **现状**：前端 `fetch()` 直连 AI API（v0.1 方案），API Key 运行时仍在前端内存中
- **建议**：
  - 在 Rust 端增加 `call_ai_api` command，统一代理所有模型请求
  - 前端只传 `modelId` + `messages`，Rust 从系统凭据读取 Key 后发起请求
  - 配合 streaming：Rust 端通过 Tauri event 逐 chunk 推送
- **实现**（2026-07-07）：完成非流式部分代理。Rust 端新增 `call_ai_api` 命令，按 provider 分流（OpenAI 兼容 / Anthropic / Gemini），通过 `keyring` 从系统凭据读 Key 后发起 `reqwest` 请求，API Key 全程不离开 Rust 进程。前端 `aiGateway.ts` 新增 `callAiApiViaRust` 封装 `invoke`，`callOpenAICompatible` / `callAnthropic` / `callGemini` 三个非流式函数优先走 Rust 代理，失败或浏览器环境自动回退到原 `fetch` 路径。`generateText` / `generateJson` / `testConnection` 与 Agent 分析任务（`runAgentAnalysis` 用 `generateJson`）均自动受益。流式 `streamChat` 暂保留前端 fetch + SSE 直连（2.4 滑动窗口已完成，流式非本项关注点）。

- [ ] #### 2.8 Prompt 模板 / 分析策略

- **NextChat 做法**：预设角色 prompt（Mask），一键切换
- **现状**：Agent 分析 prompt 硬编码在 `aiGateway.ts` 中，用户无法自定义
- **建议**：
  - 允许用户在记忆库或设置页保存自定义分析模板
  - Agent 创建时可选用不同分析策略（"短线视角""长线价值""技术面为主"）
  - 提供 2-3 个内置模板，用户可在此基础上修改

- [~] #### 2.9 键盘快捷键

- **NextChat 做法**：Enter 发送、Shift+Enter 换行、Ctrl+K 搜索
- **现状**：✅ 已实现 Enter 发送 / Shift+Enter 换行（ChatPanel.tsx#L199-L204）；⬜ Ctrl+N / Ctrl+Shift+R / Ctrl+F 未实现
- **建议**：
  - ✅ Enter 发送消息
  - ✅ Shift+Enter 换行
  - ⬜ Ctrl+N 新建会话（依赖多会话管理 2.6）
  - ⬜ Ctrl+Shift+R 手动运行 Agent
  - ⬜ Ctrl+F 搜索聊天记录（依赖搜索功能 2.10）

- [ ] #### 2.10 聊天记录搜索与筛选

- **现状**：无搜索功能，消息多了找不回来
- **建议**：
  - 关键词搜索当前会话
  - Agent 执行记录按日期/状态筛选
  - 消息按类型（用户/助手/Agent/系统）过滤

### ℹ️ 关于 MCP / 行情工具的决策（2026-07-07）

用户问股票行情时，AI 需要拿到实时数据。评估了三种方案：

| 方案 | 改动量 | 模型兼容性 | 状态 |
|---|---|---|---|
| 轻量级上下文注入 | 3-4 文件 | 所有模型（含 DeepSeek） | ✅ 已采用 |
| 标准 Function Calling | 6-8 文件 | 需模型支持 FC | ⏸️ 暂不采用 |
| 完整 MCP 协议 | 10+ 文件 | 需 MCP runtime | ⏸️ 暂不采用 |

**当前方案**：`parseUserInput` 阶段让 AI 识别用户问的股票（`queriedSymbols` 字段），`streamChat` 前调 `getBatchQuotes` 拿行情塞进 systemPrompt。所有 OpenAI 兼容模型都能用，无需模型支持 Function Calling。

**未来演进**：当需要让 AI 主动调工具（如查新闻、查公告、查财报）时，再升级到 Function Calling 或 MCP。届时本方案可作为"系统注入上下文"的 fallback 保留。

## 5. 与 a-stock-data 对标：AI 分析体验差距分析

> 日期：2026-07-08
> 来源：对照 [a-stock-data](https://github.com/simonlin1212/a-stock-data)（1,020+ Stars，A 股全栈数据工具包）进行差距分析
> 定位：找出 AI 分析能力的核心短板，明确数据层升级路径

### 5.1 定位差异

a-stock-data 是一个**面向 AI 智能体的 A 股数据中间件**。它以单一 `SKILL.md` 文件交付，整合 13 个上游数据源为 40 个标准化端点，供 Claude Code 等 AI 助手通过自然语言直接调用。它不提供独立 GUI 或自有 AI 模型。

本项目的定位是**有 GUI 的个人投资工作站**，以桌面应用形态运行，自带多模型 AI 聊天 + Agent 调度 + 持仓管理。两者的关系是**互补而非竞争**——a-stock-data 提供的恰好是本项目当前最大的短板。

### 5.2 数据层差距（AI 分析体验的根基）

当前本项目数据源仅 **1 个**（新浪财经实时报价），a-stock-data 有 **13 个**。这意味着 AI 在做持仓分析时能获取的信息维度差距巨大：

| 数据维度 | 本项目 | a-stock-data | 对 AI 分析的影响 |
|---------|--------|-------------|-----------------|
| 实时报价 | ✅ 新浪 | ✅ 新浪/腾讯/同花顺 | — |
| K 线（含 MA） | ❌ | ✅ 多周期 K 线+MA5/10/20 | **AI 无法做技术面分析** |
| 五档盘口 | ❌ | ✅ | AI 无法评估流动性 |
| PE/PB/市值 | ❌ | ✅ 腾讯财经 | **AI 无法做估值判断** |
| 季报财务数据 | ❌ | ✅ 37 字段季报 | **AI 无法做基本面分析** |
| 财报三表 | ❌ | ✅ 利润表/资产负债表/现金流量表 | AI 无法评估盈利质量 |
| F10 资料 | ❌ | ✅ 九大类 | 缺少公司治理/股东信息 |
| 一致预期 EPS | ❌ | ✅ 同花顺 | AI 无法做前瞻估值 |
| 北向资金 | ❌ | ✅ 东财 | **AI 看不到外资动向** |
| 融资融券 | ❌ | ✅ 东财 | AI 无法判断杠杆情绪 |
| 大宗交易 | ❌ | ✅ 东财 | 缺少主力动向数据 |
| 股东户数 | ❌ | ✅ 东财 | **无法判断筹码集中度** |
| 龙虎榜 | ❌ | ✅ 东财 | 缺少游资追踪 |
| 涨停板分析 | ❌ | ✅ 涨停池/炸板池/连板梯队 | 缺少短线情绪 |
| 行业排名 | ❌ | ✅ | 无法做行业内横向对比 |
| 机构研报 | ❌ | ✅ 东财/iwencai | **AI 无机构视角** |
| 互动易问答 | ❌ | ✅ 巨潮 cninfo | 缺少独家信源 |
| 热榜/人气 | ❌ | ✅ 同花顺/东财 | AI 无法感知市场情绪 |
| 公告检索 | ❌ | ✅ 沪深北交易所 | 缺少事件驱动分析 |
| 解禁预警 | ❌ | ✅ | 缺少风险事件提醒 |
| ETF 期权 | ❌ | ✅ T 型报价/希腊字母/IV | 无法做对冲研判 |

### 5.3 AI 分析能力差距

| 能力 | 本项目 | a-stock-data 生态 | 差距描述 |
|------|--------|-------------------|---------|
| 持仓诊断 | 单一模型给建议 | 7 位 AI 分析师多空辩论 (TradingAgents-astock) | 单视角 vs 多视角辩论 |
| 分析维度 | 仅价格+盈亏 | 估值/技术/资金/情绪/筹码 5 维度 | 维度数量 1 vs 5 |
| 调研深度 | 无结构化流程 | 4 套调研流水线（30s-2min） | 无标准投研 SOP |
| 选股能力 | 无 | 多因子筛选+题材归因 | 无法主动发现机会 |
| 回测能力 | 无 | 可接 backtrader/qlib/vnpy | 无法验证策略 |
| 对冲研判 | 无 | ETF 期权希腊字母+IV | 无法做风险对冲 |
| 技术指标 | 无 | MA/MACD/RSI/KDJ/布林带 (stockstats) | 无技术面信号 |
| 情绪感知 | 无 | 涨停板/热榜/人气排名 | 无法感知市场温度 |

### 5.4 具体优化建议（数据层专项）

#### 🔴 Tier 1 — AI 分析质量的生命线（对齐 Tier 1 优先级）

- [ ] #### 5.4.1 引入 K 线数据源

  当前 AI 看到的信息是"贵州茅台现价 1680，你成本 1650，浮盈 1.8%"，然后让 AI 给建议。这等于让一个分析师蒙着眼睛做判断。

  **建议**：
  - 接入通达信 mootdx（TCP 7709，无封 IP 风险，a-stock-data 第 1 优先数据源）
  - 在 Rust 端新增 `get_kline(symbol, period)` 代理命令（mootdx 是 Python 库，需通过 Python 子进程或 HTTP 桥接，或改用腾讯/百度 K 线 API）
  - 短期内可走**腾讯财经 K 线 API**（HTTP，无封 IP 风险，Rust 端直接代理）
  - Agent 分析 prompt 中注入最近 60 日 K 线数据 + MA5/MA10/MA20
  - AI 便能输出"股价目前在 20 日均线下方，短期趋势偏弱"这类有数据支撑的判断

- [ ] #### 5.4.2 引入基本面数据

  没有 PE/PB/ROE，AI 无法判断"贵不贵"。当前 AI 只会说"涨了"或"跌了"——这是个计数器，不是分析师。

  **建议**：
  - 接入腾讯财经基本面 API（PE/PB/市值/换手率）
  - 接入同花顺一致预期 EPS（用于前瞻估值）
  - Agent 分析 prompt 中注入 PE 分位、PB、ROE 等核心指标
  - AI 便能输出"当前 PE 35 倍，处于历史 80% 分位，估值偏贵"

- [ ] #### 5.4.3 引入资金面数据

  **建议**：
  - 接入东方财富北向资金流向（有封 IP 风险，需做限流）
  - 接入股东户数变化（筹码集中度，a-stock-data 已实现）
  - 优先做"北向资金净流入/流出"和"股东户数变化"这两个对 AI 分析最有价值的指标

- [ ] #### 5.4.4 扩展行情数据源 fallback

  **现状**：仅有新浪一个数据源。一旦新浪挂掉或封 IP，行情功能完全瘫痪。
  **建议**：
  - 增加腾讯财经作为第二数据源（HTTP 简单，无封 IP 风险）
  - 实现"新浪 → 腾讯"自动 fallback
  - Agent 分析中的趋势数据（MA/涨跌幅）走更稳定的数据源

#### 🟡 Tier 2 — AI 分析深度升级

- [ ] #### 5.4.5 多维度 AI 分析报告

  当前 Agent 分析输出结构（`AgentAnalysisOutput`）只有 `marketOverview` / `risks` / `opportunities` / `suggestion`，维度单一。

  **建议**：扩展分析输出为 5 维度：
  - **技术面**：趋势判断 / 均线位置 / 量价关系 / 支撑压力位
  - **基本面**：估值分位 / 盈利能力 / 成长性 / 财务健康度
  - **资金面**：主力动向 / 北向态度 / 筹码变化
  - **情绪面**：市场热度 / 题材轮动 / 舆情倾向
  - **风险面**：回撤风险 / 集中度风险 / 事件风险

  每个维度独立评分（1-10），汇总为综合建议。用户可配置关注哪些维度。

- [ ] #### 5.4.6 多模型辩论机制

  **参考**：TradingAgents-astock 的 7 位 AI 分析师多空辩论架构
  **建议**：
  - 利用已有的 9 厂商多模型支持
  - 对同一持仓同时让 2-3 个模型各自独立分析
  - 汇总各模型观点，标注一致性和分歧点
  - 例如：Claude 看多（置信度 0.7）、DeepSeek 看空（置信度 0.6）→ 提示用户"AI 观点存在分歧，建议谨慎"

- [ ] #### 5.4.7 调研流水线

  **参考**：a-stock-data 的 4 套调研流水线
  **建议**在 Agent 中增加预设分析模板：
  - **快速估值**（30 秒）：实时价 → 一致预期 EPS → 前向 PE / PEG / 消化年数
  - **深度调研**（2 分钟）：机构覆盖 → 估值 → 概念 → 资金流向 → 龙虎榜 → 解禁 → 两融

  - **事件速览**（30 秒）：最新公告 → 研报 → 互动易问答 → 新闻

### 5.6 关于直接集成 a-stock-data SKILL.md 的评估

a-stock-data 的交付形式是 `SKILL.md`（Markdown + 内嵌 Python 代码块），可直接作为 Claude Code 的 skill 使用。但本项目是 Tauri 桌面应用，集成方式不同：

| 集成方案 | 优点 | 缺点 | 推荐度 |
|---------|------|------|--------|
| Rust 端直接代理 HTTP API | 与现有架构一致、无额外依赖 | 需逐个实现端点、工作量大 | ⭐⭐⭐⭐⭐ 推荐 |
| Rust 调用 Python 子进程 | 可复用 a-stock-data 全部 40 端点 | 用户需装 Python + pip 依赖、维护成本高 | ⭐⭐⭐ 备选 |
| 前端通过 Tauri shell 调 Python | 改动最小 | 同样需 Python 环境、错误处理复杂 | ⭐⭐ 不推荐 |

**推荐方案**：按 P0→P3 优先级，在 Rust 端逐步代理最关键的 8-10 个 HTTP API 端点（腾讯 K 线、腾讯 PE/PB、东财北向资金、东财股东户数等）。这些 API 都是纯 HTTP GET，Rust `reqwest` 可以胜任，不需要 Python 依赖。当需要更复杂的数据（如季报 37 字段、F10 九大类）时，再评估是否引入 Python 桥接层。

## 7. 一句话总结（更新版）

本项目的核心壁垒是「股票投研 Agent + 结构化持仓管理 + Windows 桌面 GUI」，a-stock-data 的优势在「13 源 40 端点的全维度 A 股数据」。优化策略：**数据层按 P0→P3 优先级逐步补齐（让 AI 从"瞎子"变成"全维分析师"），AI 分析层从单模型单维度升级为多模型多维度辩论，通用体验层保持 NextChat 对标节奏**。三者并行推进，不做成另一个通用聊天客户端，也不做成纯命令行数据工具。
